# niuma-cli 开发文档

## 1. 产品定位

`niuma-cli` 是一个面向开发者日常工作记录的命令行工具，用本地 SQLite 保存项目、待办、进度、日报等数据，并通过 AI 将零散流水账整理成适合汇报的日报、周报或复盘材料。

核心目标：

- 快速记录项目、任务和工作进展。
- 自动把完成的 Todo 物化为 Progress 流水账。
- 支持标签、计时、统计等结构化能力。
- 使用大模型把真实工作记录润色为专业日报。

## 2. 数据与配置

### 2.1 默认数据目录

CLI 默认在用户家目录创建隐藏目录，避免版本更新导致数据丢失。

- macOS / Linux：`~/.niuma/niuma.db`
- Windows：`C:\Users\<用户名>\.niuma\niuma.db`

### 2.2 配置命令

| 命令 | 作用 |
| --- | --- |
| `niuma config get db.path` | 查看当前 SQLite 数据库绝对路径。 |
| `niuma config set db.path "<路径>"` | 手动指定 SQLite 数据库存储位置，便于跨设备同步。 |
| `niuma config reset` | 恢复默认配置，将数据库路径重置为 `~/.niuma/niuma.db`。 |

### 2.3 状态命令

| 命令 | 作用 |
| --- | --- |
| `niuma --help` / `niuma -h` | 打印全部可用命令、参数说明和使用示例。 |
| `niuma status` | 查看当前数据库路径、数据库连接状态、今日已完成和未完成 Todo 统计。 |

## 3. 核心业务命令

### 3.1 项目管理

项目用于承载 Todo 和 Progress 的业务归属。

| 命令 | 作用 |
| --- | --- |
| `niuma project create <name>` | 创建项目，例如 `niuma project create 电商系统`。 |
| `niuma project list` | 查看本地数据库中记录的全部项目。 |

### 3.2 Todo 管理

| 命令 | 作用 |
| --- | --- |
| `niuma todo add "<内容>"` | 创建待办任务。未指定项目时，可通过交互式下拉选择关联 Project，也允许不关联。 |
| `niuma todo done <id>` | 将指定任务标记为完成，写入 `completed_at`，并自动向 `progress` 表插入记录。 |
| `niuma todo list [date]` | 按项目分组展示指定日期的未完成 Todo、当天新建 Todo 和当天完成 Todo。 |

`[date]` 使用 `YYYY-MM-DD` 格式，不传时默认当天。

`todo done` 自动插入的 Progress 默认内容：

```text
【已完成 Todo】<Todo 内容>
```

### 3.3 Progress 流水账

| 命令 | 作用 |
| --- | --- |
| `niuma progress log "<标题>"` | 手动记录一条工作进度，可交互式选择关联项目。 |
| `niuma progress new "<标题>"` | 预录或补录一条工作进度，语义等同于 `log`，更适合带业务时间的记录。 |
| `niuma progress modify <id>` | 修改标题、内容、标签、项目归属或业务时间。 |
| `niuma progress list [date]` | 按业务时间顺序展示指定日期一整天的工作流水账。 |
| `niuma progress show <id>` | 查看单条 Progress 的完整内容和审计时间。 |

`progress log` 和 `progress new` 支持 `--content`、`-p/--project-id`、`-t/--tag`，并支持 `--data`、`--date`、`--at` 指定业务时间。业务时间用于预录、补录、列表归档、日报和统计；`created_at` 仅表示记录创建审计时间。

`progress list` 需要同时包含手动记录和 `todo done` 自动生成的记录，并按 `happened_at` 业务时间归档。

## 4. 标签体系

标签用于把 Todo 和 Progress 结构化，便于后续统计与日报生成。

### 4.1 标签输入方式

快捷参数模式适合熟练用户或 AI Agent 操作：

```bash
niuma todo add "修复登录页验证码报错" -t Bug
niuma progress log "和前端联调支付接口" -t Feature
niuma progress new "补录支付接口联调" --data "2026-05-21 23:30" -t Feature
niuma progress log "参加半年度绩效复盘会" -t Meeting
```

交互式模式适合手动输入：当用户没有提供 `-t` 或 `--tag` 参数时，终端弹出单选列表。

```text
$ niuma todo add "优化首页加载速度"

? 请选择任务类型: (Use arrow keys)
> Feature (新功能)
  Bug (缺陷修复)
  Refactor (代码重构)
  Meeting (会议/沟通)
  Research (技术调研/排查)
```

建议内置常用标签：

- `Feature`
- `Bug`
- `Refactor`
- `Meeting`
- `Support`
- `Research`

### 4.2 标签配置命令

| 命令 | 作用 |
| --- | --- |
| `niuma config tags list` | 查看当前系统支持的所有标签。 |
| `niuma config tags add <tag_name>` | 新增自定义标签。 |
| `niuma config tags del <tag_name>` | 删除自定义标签。 |
| `niuma config tags rename <old_name> <new_name>` | 重命名标签，并同步迁移历史数据。 |

### 4.3 标签数据策略

V1 阶段可采用轻量方案：Todo 和 Progress 表直接保存 `tag` 字符串，而不是保存 Tag ID。

因此标签重命名时必须同步执行：

1. 修改配置或配置表中的标签列表。
2. 批量更新 Todo 和 Progress 历史数据里的旧标签值。

删除标签前必须先检查是否存在引用。如果标签已经被 Todo 或 Progress 使用，应拦截删除并提示用户先迁移。

示例提示：

```text
标签 [摸鱼] 正在被 3 个任务使用，无法删除。请先执行 rename 迁移或使用后续的强制转移能力。
```

## 5. 日报生成

### 5.1 命令

```bash
niuma daily generate [date]
```

`[date]` 可选，不传时默认当天。

### 5.2 核心流程

1. 从 SQLite 按 `happened_at` 业务时间读取指定日期的 Progress 流水账。
2. 补充读取当天完成的 Todo。
3. 将原始记录组装为 Prompt。
4. 调用大模型 API，把零散工作记录整理成结构清晰、用词专业、适合汇报的日报。
5. 在终端输出日报。
6. 将生成结果保存到 `dailies` 表，便于后续周报、月报或年终总结复用。

### 5.3 Prompt 方向

```text
请将以下程序员的工作流水账，转化成一份结构清晰、用词专业、适合向上级汇报的日报文本。
```

## 6. 扩展模块

### 6.1 专注计时

命令设计：

| 命令 | 作用 |
| --- | --- |
| `niuma todo focus <id>` | 开始专注处理某个任务，终端进入计时或番茄钟状态。 |
| `niuma todo stop` | 结束当前专注计时。 |

结束专注时自动向 `progress` 表插入定量记录，例如：

```text
【专注】耗时 45 分钟推进了该任务
```

日报生成时，AI 可以基于这些定量数据生成更有说服力的描述，例如“今日累计专注业务研发 4.5 小时，高效解决多个核心阻塞点”。

### 6.2 Chill 非工作状态记录

命令设计：

| 命令 | 作用 |
| --- | --- |
| `niuma chill start "<内容>"` | 开始记录一段非工作或等待状态。 |
| `niuma chill end` | 结束当前 Chill 记录。 |

数据策略：

- 数据库中为 Chill 记录打上特殊隐藏标签。
- `daily generate` 读取后由 AI 转化为更适合汇报的表达。

示例：

```text
原始记录：等后端配环境等了 1 小时，刷了会儿掘金。
日报表达：由于上游依赖环境处于配置阶段，期间主动进行技术调研与行业前沿技术横向对齐，确保后续接入无缝衔接。
```

### 6.3 统计与复盘

命令设计：

```bash
niuma stats [week|month]
niuma daily generate --week
```

统计内容：

- 指定周期内完成了多少个 Bug、Feature 等标签任务。
- 哪个项目投入最多。
- 终端用 ASCII 柱状图或表格展示生产力趋势。
- 一键生成周报，把一周的数据库存根聚合成可汇报文本。

## 7. 建议数据模型

### 7.1 projects

| 字段 | 说明 |
| --- | --- |
| `id` | 项目 ID。 |
| `name` | 项目名称。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

### 7.2 todos

| 字段 | 说明 |
| --- | --- |
| `id` | Todo ID。 |
| `project_id` | 关联项目 ID，可为空。 |
| `content` | Todo 内容。 |
| `tag` | 标签字符串。 |
| `status` | 任务状态，例如 `pending` / `done`。 |
| `created_at` | 创建时间。 |
| `completed_at` | 完成时间，可为空。 |

### 7.3 progress

| 字段 | 说明 |
| --- | --- |
| `id` | Progress ID。 |
| `project_id` | 关联项目 ID，可为空。 |
| `todo_id` | 来源 Todo ID，可为空。 |
| `title` | 流水账标题。 |
| `content` | 流水账详情，可为空。 |
| `tag` | 标签字符串。 |
| `source` | 来源，例如 `manual` / `todo_done` / `focus` / `chill`。 |
| `started_at` | 开始时间，可为空。 |
| `ended_at` | 结束时间，可为空。 |
| `happened_at` | 业务发生时间，用于预录、补录、列表、搜索、日报和统计归档。 |
| `created_at` | 创建审计时间。 |
| `updated_at` | 更新时间。 |

### 7.4 dailies

| 字段 | 说明 |
| --- | --- |
| `id` | 日报 ID。 |
| `date` | 日报日期。 |
| `content` | AI 生成后的日报内容。 |
| `raw_context` | 生成日报时使用的原始记录快照。 |
| `created_at` | 创建时间。 |

### 7.5 config 或配置文件

| 字段 | 说明 |
| --- | --- |
| `db.path` | SQLite 数据库路径。 |
| `tags` | 用户配置的标签列表。 |
| `llm` | 大模型配置，后续接入 AI 时再定义。 |

## 8. 开发阶段规划

### V1.0 核心骨架

目标：完成可用的本地工作记录闭环。

范围：

- Project 创建与列表。
- Todo 创建、完成与列表。
- Progress 手动记录与列表。
- SQLite 存储。
- `status` 和基础配置查看。

验收标准：

- 用户可以创建项目。
- 用户可以创建并完成 Todo。
- `todo done` 会自动生成 Progress。
- 用户可以查看当天 Todo 和 Progress。
- 数据默认落在 `~/.niuma/niuma.db`。

### V1.1 标签注入

目标：让记录可统计、可分类。

范围：

- `-t` / `--tag` 快捷参数。
- 未传标签时的交互式选择。
- `config tags list/add/del/rename`。
- 标签删除保护。
- 标签重命名历史数据迁移。

验收标准：

- Todo 和 Progress 都能保存标签。
- 已被引用的标签不能直接删除。
- 标签重命名后历史 Todo 和 Progress 同步更新。

### V1.2 AI 日报

目标：接入大模型，生成日报。

范围：

- `niuma daily generate [date]`。
- 读取当天 Progress 和完成 Todo。
- 组装 Prompt。
- 调用大模型 API。
- 保存生成结果到 `dailies` 表。

验收标准：

- 无记录时给出清晰提示。
- 有记录时能生成日报并打印到终端。
- 生成内容可被持久化查询。

### V2.0 自动化增强

目标：减少用户手动记录成本，并支持复盘。

范围：

- `todo focus` / `todo stop` 专注计时。
- `chill start` / `chill end` 非工作状态记录。
- `stats [week|month]` 统计图表。
- `daily generate --week` 周报生成。
- 后续可扩展 webhook 推送到飞书或钉钉。

验收标准：

- 专注计时结束后自动生成 Progress。
- Chill 记录能被 AI 转化为适合汇报的表达。
- 周/月统计能按标签和项目聚合。

## 9. 实现原则

- 优先实现 V1.0 的最小闭环，不要一次性把所有模块写完。
- CLI 命令采用“动词 + 名词”或“名词 + 动词”的稳定规范，避免同类能力多套入口。
- 本地数据操作默认使用 SQLite，先保证离线可用。
- 自动化副作用必须可追踪，例如 `todo done` 自动生成的 Progress 需要标记 `source=todo_done`。
- 标签轻量存储可以先用字符串，但重命名和删除必须处理历史数据一致性。
- AI 能力必须保留原始记录快照，方便用户追溯生成依据。
- 终端输出应优先清晰、短句、可复制，统计输出可以使用 ASCII 表格或柱状图。
